## [Unreleased]
### Features
- **Tap Loop alone (drum track) unlatches all latched repeats on that track.** When a drum track has Rpt1 or Rpt2 lanes latched, you can now clear them with a single tap of the Loop button — provided no pads/lanes are held at the moment of press, the active track is in drum mode, and the tap is short (no length gesture, no held-pad latching). Holding pads + tapping Loop still latches (unchanged); long-pressing Loop still does nothing in this branch (unchanged); tap-alone on a melodic track is a no-op. Drains via `pendingDefaultSetParams` so the unlatch lands cleanly even alongside concurrent audio-thread traffic. (Eligibility for Rpt1 treats `drumRepeatHeldPad` as a *fresh* hold only when `drumRepeatLatched` is false — once a rate pad is latched, that field stores the latched-pad reference rather than a physical press, so the unlatch tap-alone must still qualify.)

### Fixes
- **Coalescing remediation, transitive sweep.** Follow-up to the 2026-05-20 bank-reset fix. Audited every JS function that calls `set_param` and traced callers back to MIDI handlers; migrated the helpers that were still firing synchronously to `pendingDefaultSetParams` (one-per-tick drain). Affected gestures: Copy / Cut clip (Session and Track View), Copy / Cut row, Copy / Cut drum lane, Copy step, Clear step, Clear row, Snapshot save / load / delete (Mute + step), Scene launch, Live Merge arm / stop. None had a confirmed user-visible bug — preemptive defense against the same coalescing class.
- **Bank reset routing on CC PARAM bank fixed for drum tracks.** Delete+jog while on the CC PARAM bank used to fire `tN_lL_repeat_groove_reset` (and pop "RPT GROOVE RESET") on a drum track in Rpt1 / Rpt2 mode — because the bank-6 CC-clear branch was nested inside the melodic path and never reached. The CC PARAM check now runs first regardless of pad mode or drum perform mode, so a drum track on CC PARAM properly clears its CC automation (and shows "CC AUTO CLEAR"). Repeat-groove reset still fires when actually on a drum bank in Rpt mode.
- **CC PARAM OLED values clear after CC automation clear.** The clear handler reset `trackCCAutoBits` + `trackCCLiveVal` but not `trackCCVal`, which is what the bank 6 OLED draw actually reads to display the per-knob value. Result: automation cleared on DSP, audio stopped firing CC, but the OLED kept showing the pre-clear knob values. `trackCCVal` is now zeroed alongside the rest. The DSP-side `cc_auto_clear` handler was correct; this was a JS-side display state miss.
- **Bank reset / param reset now actually reach DSP.** Delete+jog (single bank reset), Shift+Delete+jog (all real-time FX banks), Delete+jog in Rpt mode (REPEAT GROOVE reset), and Delete+jog on the CC PARAM bank (CC automation clear) all showed the success popup on the OLED but left DSP playback unchanged. Cause was the same as the bake-confirm coalescing fix earlier in this batch — the reset `set_param` push fired synchronously from inside the jog CC handler, competed with the same audio buffer's MIDI delivery, and was silently coalesced away. All six reset call sites (`tN_pfx_reset`, `tN_lL_pfx_reset`, `tN_<bank>_reset`, drum-lane single-bank reset, `tN_lL_repeat_groove_reset`, `tN_cc_auto_clear`) now route through `pendingDefaultSetParams` (one-per-tick drain). Queue order preserved so the `delay_level=127` post-reset override still lands after the reset zeros it.
- **Sample tap in Session View no longer opens the clip bake dialog.** Clip bake is a Track View action — opening it from Session View opened a per-track dialog targeted at whatever happened to be the active track, which was almost never what you wanted in a launcher context. Bare Sample tap in Session View is now a no-op; Sample still works as a modifier for scene bake (hold Sample + tap a scene row). Track View behavior is unchanged.
- **Note duration hold+tap now extends THROUGH the tapped step.** Holding step 0 and tapping step 4 used to set the note's gate to end AT step 4 (4 steps of duration); it now extends through step 4 (5 steps, ending at the boundary of step 5) so the visible "lit" range matches the sounding length. Same off-by-one fix applied to both the drum-lane step-edit handler and the melodic clip step-edit handler.
- **Clear Session also resets channel / pad octave / route / looper to defaults.** Companion to the 2026-05-19 `pad_mode`/`active_drum_lane`/`drum_perform_mode` fix in the same `state_load` reset loop. Per-track `channel`, `pad_octave`, `pfx.route`, and `pfx.looper_on` all used to keep their pre-clear DSP values after Clear Session (JS mirrors were reset but never pushed to DSP); for valid `v=28` saves, `seq8_load_state` still overwrites these resets with the persisted values, so normal set-loading is unchanged. Drum-lane routes follow the per-track route reset so default routing (tracks 1-4 → Move, 5-8 → Schwung) is restored across the board.

- **Drum repeats during count-in + first-fire Repeat Sync + true sub-step recording.** Holding a Rpt1 / Rpt2 pad through count-in now sounds the repeats audibly through the click, then fires seamlessly into the recorded clip at the loop window start — same warm-up affordance ARP IN already has. A new per-track Repeat Sync toggle (ALL LANES bank K6, default ON) replaces the old InQ-tied first-fire snap: when on, the first repeat-fire of a press waits for the next rate-grid boundary (anchored to a free-running clock that resets at transport play / count-in fire) regardless of whether transport is stopped or playing; when off, the first fire is instant. And when recording with InQ on, every repeat fire now writes its actual sub-step offset into the clip instead of collapsing to the step boundary — a 1/32 rate played against a 1/16 step grid records as a true 1/32 cadence with adjacent-step notes carrying their own offsets, exactly the timing you heard. DSP state bumped to v=28; user clips will reset on first boot of this build per the pre-public no-migration posture.

### Fixes
- **Side clip button now focuses the clip on press, not on the legato boundary.** In Track View, pressing a side clip button while transport was playing used to queue the legato switch but leave the OLED + pads + bank params stuck on the previously-playing clip until DSP actually crossed the loop boundary. So the clip you just pressed was "active" musically but still invisible — you had to wait for the switch to land before you could edit it. JS now updates `trackActiveClip` immediately on every side-button press regardless of transport state; `pollDSP` keeps the value in sync if DSP's boundary lands later. Empty clips are launchable too — focus + launch happen together.
- **Stuck live notes when touching the SEQ ARP / ARP IN Steps knob mid-hold.** Holding a pad on a melodic track then touching K5 (Steps Mode) to bring up the per-step level overlay used to hang the note until the next pad press of the same pitch. The padmap push at modal-mute transitioned all 32 pads to `0xFF`, which made DSP's `on_midi` early-return on `pitch == 0xFF` before reaching `live_note_off` — so the release MIDI was dropped on the floor. Fix is two-part: (1) JS now keeps the real per-pad pitch values when `pad_dispatch_muted` is set (in track view; session view still publishes all-0xFF), so the note-off has a valid pitch to dispatch against; (2) DSP gates `live_note_on` on `!pad_dispatch_muted` (so new presses during the overlay are still suppressed) while `live_note_off` always passes through. Releases of in-flight live notes are now reliable across every modal-mute transition.
- **SEQ ARP Steps Mode knob now takes effect on the first turn — and Off actually bypasses the grid.** The Steps Mode `set_param` from the bank-knob handler was getting eaten by per-buffer coalescing against the same audio buffer's MIDI dispatch — first turn from Off to Mute silently did nothing, and going back to Off didn't restore bypass either. Same diagnostic family as the padmap drop and the drum→Keys Mode flip: queued `set_param` losing to same-buffer MIDI. Fix routes the Steps Mode push through the existing `pendingDefaultSetParams` queue (one-per-tick drain). DSP also picks up a defensive `level = 4` normalization at all three step-velocity read sites (SEQ ARP runtime, ARP IN runtime, bake) so even if a future code path leaks a non-Off `level` read while `steps_mode == 0`, the grid value can't influence playback.
- **Clear Session now actually resets drum-mode tracks back to Keys (except track 1).** Per the user spec, Clear Session should start the set "as if the session never existed" — only track 1 starts in drum mode, everything else in Keys. In practice, any track the user had set to drum mode stayed in drum mode after the clear: the LEDs and JS state correctly flipped to Keys defaults, but DSP's audio thread kept classifying the track as drum, ghost-firing right-half pad presses through the drum vel-zone classifier (and disabling melodic pad dispatch). Root cause was in `seq8_load_state`: on a version mismatch (Clear Session writes `{"v":0}` to force the wipe), the function deleted the file and returned WITHOUT touching in-memory track state. JS thought it had cleared everything, but `tr->pad_mode` / `tr->active_drum_lane` / `tr->drum_perform_mode` for previously-drum tracks all persisted in the DSP instance. Fix: the DSP `state_load` set_param handler's existing per-track reset loop (which already resets transport state, clip data, drum lanes, drum-repeat defaults) now also resets `pad_mode` / `active_drum_lane` / `drum_perform_mode` to the same defaults `create_instance` uses. For normal set-switching with a valid `v=28` file, `seq8_load_state` overwrites these resets with the saved values, so set-load behavior is unchanged. For Clear Session (`v=0` file gets deleted), the defaults stand and JS's `restoreUiSidecar` first-run path re-pushes `t0_pad_mode=DRUM` so track 1 still wakes up in drum mode as expected. Other track-config fields (`channel`, `pfx.route`, `pfx.looper_on`, `pad_octave`) likely have the same drift on Clear Session but the user hasn't reported them — filed in TODO.md against the same code path.
- **Drum→Keys Mode flip from the Track menu now actually takes effect on DSP.** Switching a track from Drums back to Keys via Global Menu → Track → Mode used to leave the DSP audio thread still classifying the track as a drum track — the LEDs and JS state flipped correctly, but pressing pads on the right 4x4 would ghost-fire the previously-pressed drum lane's note at vel-zone velocities, and the actual `tN_pad_mode='0'` set_param never reached DSP. The mechanism is in the same family as the zombie-clip fix (commit 2a00073) — same-buffer set_param interference — but the *exact* mechanism is different. Zombie-clip-B was a queued `_clear` set_param losing to a same-on_midi-handler synchronous fan-out (fixed with `clearDrainHold`). This bug is **multiple synchronous `tN_*` set_params for the same track in the same tick silently drop the first one**, even when the keys are different. The leaving-DRUM path was firing 4 of them back-to-back from the same JS callback (`tN_pad_mode='0'`, `tN_active_drum_lane='0'`, `tN_drum_perform_mode='0'`, then `tN_padmap` via `computePadNoteMap`); `pad_mode` was the casualty every time. The entering-DRUM path escapes the same interference by accident — it fires `syncDrumLanesMeta` / `syncDrumLaneSteps` / `syncDrumClipContent` get_params between the `pad_mode` and `padmap` pushes, and the get_param round-trips to the audio thread act as a flush barrier. Fix: in `applyTrackConfig`, route the two extra DSP mirror pushes (`active_drum_lane`, `drum_perform_mode`) through the existing `pendingDefaultSetParams` queue (one-per-tick drain), and defer `computePadNoteMap` via a new `pendingPadNoteMapRecompute` flag that fires only once the queue is empty — so the leaving-DRUM sequence becomes: `tN_pad_mode='0'` (alone, tick N) → `adl='0'` (tick N+1) → `dpm='0'` (tick N+2) → `tN_padmap=<melodic>` (tick N+3, alone). Each push lands cleanly. Empirically verified end-to-end with a JS-side trace ring buffer and DSP `seq8_ilog` probes on every relevant set_param handler. The companion bug from the same parking note — Clear Session leaving drum-mode tracks stuck in drum mode after a session reset — is still parked; see `notes/drum-melodic-mode-flip-and-clear-session-drift.md`.
- **Drum vel-zone (right 4x4) pad release now sends note-off.** Tapping a right-half drum pad in NORMAL drum mode fired `live_note_on` on press but did nothing on release — `drum_pad_event`'s release branch was a deliberate no-op with a "synth ringout" comment. Destinations that expect a note-off (ROUTE_SCHWUNG, USB external, anything without a self-releasing envelope) hung the note indefinitely. The release branch now fires `live_note_off` for the active lane's note, with `pad_source_scratch = PAD_SRC_VEL_ZONE` bracketing matching the press path. Synth envelope release behavior is unchanged on destinations that release on note-off (the standard contract).
- **Zombie clips after clear are dead.** Clearing a clip (Delete + clip-button in Track View, Delete + clip-pad in Session View, Shift+Delete for full reset, or Global Menu → Clear Sess) used to leave the cleared notes audible on next play — the step grid looked empty, but transport kept firing the old notes, and toggling any step would reveal the whole pattern. Two independent bugs were combining: (1) the DSP's `state_full` get_param returned its cached serialization buffer even when the state hadn't changed since the last save, so JS's deferred-save poll kept overwriting the on-disk file with the stale pre-clear blob — making the bad state come back across reboot. (2) clearClip's queued `tN_cC_clear` set_param was getting eaten by per-buffer coalescing: clearClip runs in the on-MIDI handler and synchronously fires a dozen-plus `pfx_set` calls plus a `_length` write in the same audio buffer that the drain tried to deliver `_clear` on the next tick — and the host coalesces all set_params in one buffer down to a single survivor. `_clear` was always the loser. Fix: `state_full` returns empty when not dirty (JS skips the write); clearClip / hardResetClip set a one-tick drain hold so `_clear` / `_hard_reset` / `_drum_clear` / `_drum_reset` land in a clean buffer.
- **Pads no longer go silent on the DSP audio thread after a modifier toggle.** Commit 7ab2477 surfaced JS's `_padDispatchMutedNow()` to DSP via a 34th token appended to the `tN_padmap` `set_param` push, so that `on_midi` could gate Rpt1 / Rpt2 / vel-zone dispatch on the same modal-mute state JS uses. The wrinkle: that `set_param` channel shares its delivery queue with `shadow_send_midi_to_dsp` and coalesces same-key writes per audio buffer, so un-mute pushes (e.g. immediately after releasing Mute, or after closing the slot picker) intermittently dropped — leaving DSP stuck at `pad_dispatch_muted=1` while JS believed it had successfully pushed `0`. All pads silent on every track until the user happened to gesture a modifier (Shift / Mute / Delete) which retriggered `computePadNoteMap`. Reproduced reliably as: tap Mute → tap Mute → pads silent → tap Shift → pads back. Fix is JS-side self-heal: every 5 ticks (~50ms), JS reads DSP's `pad_dispatch_muted` via a new `get_param` reader and re-pushes the padmap when DSP and JS disagree, so a dropped push self-corrects within 50ms instead of indefinitely.
- **Belt-and-suspenders modifier blanket-clear on every co-run exit.** Both `Edit Synth` (Move-native) and `Edit Slot` (Schwung chain editor) co-runs now clear Shift / Delete / Mute / Copy / Loop / Capture / `shiftTrackLEDActive` at all four exit paths — the dAVEBOx-initiated exits plus the two SHM-reconcile paths that fire when shadow_ui or the shim dismiss the co-run externally — mirroring the resume-from-suspend pattern. No bug was empirically reproduced against this path, but it costs nothing and removes a whole class of theoretical stuck-modifier states from the co-run lifecycle.
- **Rpt2's default rate is now 1/8 across every lane.** Fresh sets and previously-untouched lanes used to start at 1/16 (the zero-init default); the rate-pad LED now lights at 1/8 on first entry into Rpt2 mode. Existing v=28 sidecars that didn't carry a per-lane rate key load with the new default.
- **Latched Rpt1 + Rpt2 lane pads stay persistently lit.** Lane pads holding a Rpt1 or Rpt2 latch now light Cyan regardless of which drum-perform mode is currently active — matches the TARP latched-chord visual language. Held-but-not-latched Rpt2 lanes also light Cyan but only while you're in Rpt2 mode (transient gesture feedback).
- **Mute silences TARP / Rpt1 / Rpt2 emission while keeping the latch alive.** Track mute used to gate the sequencer but let latched arp / repeat output keep firing through the pfx chain. The three engine fire paths now early-return when the track is effectively muted; `silence_muted_tracks` handles the sustaining tail at the moment of mute. Latch state is preserved so unmute resumes mid-phrase.
- **Transport Stop now sweeps Rpt1 + Rpt2 latches too.** The 1af9ed3 Stop sweep covered TARP only; latched drum repeats kept firing audibly through Stop until you toggled them off by hand. Stop now reuses the existing `unlatchAllTracks()` helper that the Delete + Play branch already used — TARP + Rpt1 + Rpt2 all clear together on every track.
- **Shift + bottom-row pad no longer triggers Rpt1 / Rpt2 latch on the prior track.** The Shift + pad track-switch shortcut would route the same press into DSP's drum-pad classifier, which engaged a repeat on whatever track was active *before* the switch. The padmap now carries a `pad_dispatch_muted` flag (34th token) that DSP's `on_midi` reads to skip `drum_pad_event` while any modal hold is active.
- **Record-arming during play no longer drifts TARP timing.** Pressing Record while transport was already running silenced the TARP chord — `tarp_silence()` reset `master_anchor=0`, which shifted the step index and drifted the latched pattern. The silence call now skips when transport is already playing; TARP keeps its anchor and stays locked to the grid.
- **Rpt1 + per-lane Rpt2 rates now persist across reload.** The sparse keys `t%d_drrt` (Rpt1, per-track) and `t%dl%dr2rt` (Rpt2, per-lane) save and load alongside the existing groove fields; a new `tN_drum_r2rt` batch get_param resynchronises the JS rate-pad LED highlight after state load. No state version bump — sparse keys default to 1/8 (idx 2) on absence.
- **Input Quantize is now per-track and snaps to its actual rate value.** Two coupled changes. (1) The drum InQ math at `drum_record_note_on` only quantized the tick position **within the current step**, so any InQ value coarser than 1/16 (1/8, 1/4, etc.) silently collapsed to "snap to current step boundary" — identical to the old global Input Quantize behavior, defeating the per-track granularity the ALL LANES K5 control had always claimed. The math now quantizes the **global tick position** (`base_step * TICKS_PER_STEP + base_off`) then re-derives step + off, with window-aware wrap if the snap pushes past the loop end. (2) Melodic tracks get a new per-track InQ control at **CLIP K6** with the same 9 values as the drum ALL LANES K5. The DSP recording paths (drum + melodic) share the per-track `tr->drum_inp_quant` field, so toggling either UI surface reaches the same backing value. The global `Input Quantize` menu item is removed; both DSP record paths drop their `inst->inp_quant` fallback. No state version bump — the per-track field has been saved in v=28 saves since the prior commit, just unused by melodic recording.
- **Transport Stop now unlatches TARP across every track.** Latched arp chords used to keep droning after pressing Stop — `tarp_tick` runs on a free-running clock (`arp_master_tick`) that advances even while transport is stopped, so a latched buffer would keep firing pfx-side notes audibly. Stop now sweeps every track and fires `tN_tarp_latch=0`, which invokes DSP `tarp_drop_latched()` → `tarp_silence()` (clears the held buffer and silences any sounding note). Delete + Play retains its separate role as the comprehensive panic gesture.
- **TARP latch now survives track / route / channel / MIDI-in changes.** Previously, switching to a different track (or changing route/channel/MIDI-in on the active track) silently cleared the previous track's TARP latch — both the OLED chip indicator AND the DSP held buffer, so notes stopped sounding. The cleanup was tied to the cable-2 channel-remap apply path and dated back to an earlier remap-correctness fix. The latch is per-track musical intent and now persists across these plumbing changes; only Stop transport and Delete + Play clear it deliberately.
- **Modal pad-interception fixed: pad presses no longer leak into the synth while certain dialogs and modifiers are held.** With Phase 1's audio-thread pad path active on patched Schwung, presses bypassed the JS-side modal gates and reached DSP directly — so tapping a pad while Tap Tempo was open (or while Shift / Delete / Copy / Mute / Capture / Loop were held, or in Session View, or inside the narrow ARP step-edit condition) would unintentionally fire a note. dAVEBOx now publishes an "all-pads muted" padmap (0xFF for every pad) into DSP whenever any of those modal conditions is active, via the same `tN_padmap` mechanism that already drives pad layout. Edge detection runs in `tick()` so dialog and knob-touch transitions take effect without explicit hooks. Stock-Schwung users were unaffected (their pad input was already gated in JS). Global Menu and Random dialog intentionally still pass pad presses through — user-confirmed behavior.
- **VelIn now applies to live pad monitoring on patched Schwung — and TARP arp output now respects VelIn for the first time.** With a fixed VelIn value set (e.g. 100), gently tapping a pad used to play at your live touch velocity instead of the fixed value, because Bundle 1's audio-thread pad path didn't yet apply the VelIn override (the JS line that did so at `liveSendNote` was skipped on patched Schwung). dAVEBOx now calls the existing `effective_vel(tr, raw)` helper in `on_midi` right before `live_note_on`, gated on `PAD_SRC_NORMAL` so vel-zone presses (which already substitute a zone velocity inside `drum_pad_event`) pass through untouched. Incidental, free fix: TARP arp output now respects VelIn too, because `tarp_tick` reads held-pad state populated by `live_note_on` — once VelIn is applied at the entry point, the arp inherits it (no `tarp_fire_step` code change). Recording velocity on patched Schwung is unaffected (JS pre-applies VelIn via `stepEntryVelocity` before pushing the record set_param, mirroring the live rule). Stock Schwung still uses the JS-side application in `liveSendNote` (marked PHASE-1 for the eventual final cleanup pass).
- **Velocity zone presses are audible (and record) again on patched Schwung.** Tapping a right-half drum pad to preview the active lane at a chosen velocity stopped playing through the synth after the Bundle 1 audio-thread input rewrite — the JS path that fired the preview was gated off for note events on patched Schwung, but DSP didn't yet know how to classify a right-half drum pad. dAVEBOx now classifies right-half drum pads on the audio thread (`drum_pad_event`): in NORMAL mode the press arms the vel zone and fires the active lane's note at the chosen zone velocity, and the JS-driven recording path lands the hit at the correct step (populates the per-lane press snapshot the post-Bundle-1.5 preroll filter requires). Rpt1 / Rpt2 modes are explicitly handed back to the existing JS classifier until Bundle 2C ports them — `drum_pad_event` returns "handled, don't dispatch" for the activation pads in each mode (Rpt1: right-half rate pads; Rpt2: left-half lane pads, where JS pushes `tN_drum_repeat2_lane_on` and DSP's `drum_repeat2_tick` owns the first repeat). Stock Schwung users see no difference. Adds two volatile DSP mirrors (`active_drum_lane`, `drum_perform_mode`) pushed by new `tN_active_drum_lane` / `tN_drum_perform_mode` set_params from JS at every mutation site.
- **Recording into clips with a non-zero loop start now lands inside the window.** Live presses on clips whose loop window starts past step 0 used to silently drop or land at the wrong step — every recording path (melodic note-on/note-off, drum note-on/note-off, TARP overdub, finalize-pending-notes gate closure) was unconditionally collapsing window-anchored ticks with `% (length * tps)` before insertion, which strips the `loop_start` offset and points the writes at the wrong step. Drum recording additionally clamped writes with stale `step < length` bounds that rejected every legal window position when `loop_start > 0`. All sites now treat `current_clip_tick` / on-midi slot snapshots as already-window-anchored (which they are, by construction of `current_step`) and drum bounds widened to `< (loop_start + length)`. Window-wrap gate math remains correct under unsigned arithmetic for held notes that cross the loop boundary. No behavior change for `loop_start = 0` clips.
- **ARP IN's first arp note after count-in now records on step 0.** Holding a chord through count-in with ARP IN on (any sync mode) used to leave the first audible arpeggiator note out of the clip — you'd hear it play as the loop fired, then it would be missing from playback. The fire path resets `arp_master_tick` to 0 but the ARP IN engine kept its old `master_anchor` (set during a count-in tick or a pre-count-in press), which made the first post-fire `master_pos` underflow and pick the wrong pattern step; `current_clip_tick` was also reading a stale value because the per-track tick advance recomputes it *after* `tarp_tick` runs in the same DSP tick. ARP IN's runtime now re-anchors at fire (`master_anchor = 0`, `pending_first_note = 1`) and `current_clip_tick` is primed to `loop_start * tps` before the first post-fire arpeggiator fire so the capture lands cleanly at the loop window start.
- **Delete + Play now clears every latch across every track regardless of transport state.** The "kill all the drones" escape hatch used to do its full sweep — clearing Rpt1 latch, all latched Rpt2 lanes, and the TARP latch chip across all 8 tracks — only when transport was playing. When stopped, it fired a MIDI panic but left every JS-side latch mirror in its old state, so the Rpt1/Rpt2 LEDs and TARP latch-chip indicator would lie until you toggled them off manually. The per-track unlatch sweep is now extracted into an `unlatchAllTracks()` helper called from both branches of the handler. DSP-side `tN_tarp_latch=0` was already triggering `tarp_drop_latched()`→`tarp_silence()` regardless of transport — only the JS half needed the fix.
- **Per-track octave shift now actually persists per-set, matching the manual.** MANUAL §14.2 has long listed "octave shift" as a Per-track setting that persists per set, but the `trackOctave` array was JS-only — it never made it into the sidecar, so every module reload / set switch / reboot reset all 8 tracks to oct=0. The 8-slot array now saves to `seq8-ui-state.json` (bumped to v=7) on the existing suspend/Quit/Save path and restores on entry. Older v=6 sidecars stay readable — they just keep the JS-init zeros for the new field until the user saves with the new build. Clear Session resets all 8 tracks to 0 alongside the rest of the cleared state.

- **Co-run pad colors now indicate split-UI mode.** Entering Edit Synth or Edit Slot switches drum pads to grayscale (white/grey instead of track color) and inverts melodic pad colors (root notes dim, non-root notes bright track color). Exiting co-run restores normal coloring with a full LED cache flush so no stale hardware state lingers.
- **Track overview shows active clips.** In track overview, clip letters for playing or will-relaunch clips now display inverted (black on white) so you can see at a glance which clips are live across all 8 tracks.
- **Tap tempo flash shortened to ~255 ms.** The visual feedback pulse on a tap tempo press is now roughly a quarter-second instead of the old ~210 ms — feels snappier without being invisible.
- **Clip button lighting rewritten to 4 states.** Playing clips flash between bright and dim track color at 1/8-note rate. Focused-but-not-playing clips show solid bright track color. Clips with data but not focused show dim track color. Empty clips show dark grey.
- **Octave shift while holding a note no longer causes stuck notes.** Pressing Octave +/- while pads are held now sends note-offs for all active notes before shifting, then lets the next press sound at the new octave cleanly.
- **Switching track type (drums ↔ keys) no longer leaves stale LEDs.** Changing pad mode via `applyTrackConfig` now triggers a full pad note map recompute and LED redraw on the active track.
- **2-page loop can shrink to 1 page.** When a loop is exactly 2 pages, tapping the second loop-page button now converts it to a 1-page loop — previously there was no pad gesture to get back to a single page.
- **Count-in click is no longer muted by track solo.** TARP, Rpt1, and Rpt2 engine ticks now bypass the effective-mute gate during count-in so the click and any held-pad repeats/arp sound through the pre-roll regardless of which tracks are soloed.

### Performance / UX
- **ROUTE_EXTERNAL latency jitter ~7.6× tighter on patched Schwung.** Tracks routed to USB MIDI out used to emit notes from the JS tick loop (~10.6 ms stddev timing jitter, 41 ms worst-case spread); on patched Schwung, dAVEBOx now hands each outbound MIDI packet to the shim's audio-thread ring, which drains it into the SPI mailbox just before each audio-block transfer. Measured improvement (199 note-ons): stddev 10.25 ms → **1.35 ms**, P5–P95 spread 35 ms → 3 ms, worst-case 41.2 ms → 3.3 ms. Approaches the architectural floor (~2.9 ms = one audio block). Stock-Schwung users automatically fall back to the legacy JS-tick drain — no behavior change without the patched shim.
- **Pad input rewired to audio thread (on patched Schwung):** Live pad presses are now dispatched by DSP directly from the audio-thread MIDI hook, bypassing the JS-tick → `tN_live_notes` set_param queue. Chord cohesion improves on patched Schwung (no more late notes when 3-4 pads land in the same buffer) and input latency drops. Stock-Schwung users automatically fall back to the existing JS path — no behavior change without the patched shim. Works for melodic and drum tracks, ROUTE_MOVE and ROUTE_SCHWUNG, monitoring and armed recording, octave and drum-lane-page changes.
- **Count-in pre-roll capture window tightened to the last 1/8 note.** Presses during the first 7/8 of the 1-bar count-in are now monitored only (audible but not recorded); presses in the final 1/8 note land at the start of the loop window when transport flips. Previously every press during count-in was captured, which felt sloppy — warm-up taps would leak into step 0. The capture filter lives in DSP (`on_midi`), so it's enforced regardless of JS tick rate. On stock Schwung (no audio-thread pad input) the prior JS-side count-in capture continues to apply, unchanged.
- **ARP IN plays through the count-in.** Holding a chord during the 1-bar count-in click now sounds the arpeggiator immediately instead of waiting until recording fires — you can hear the pattern you're about to record. The ARP IN engine ticks alongside the count-in metronome from the moment you press; at count-in fire the engine cleanly silences any in-flight arp notes (queued note-offs reschedule to fire on the next DSP buffer — no broadcast MIDI panic, so the first audible loop step isn't cut off) and re-anchors so the first recorded fire lands on step 0 of the loop window.
- **ARP IN with `Sync = Off` captures arp output during count-in pre-roll.** With sync off the engine fires the moment you press, so chord presses that arrive in the last 1/8 note of count-in (or any earlier press still held into that window) record their arp output onto step 0 of the loop. Sync = On doesn't need this — it aligns to the rate grid and the first post-fire fire is already what lands on step 0. Direct (non-ARP-IN) pad-press pre-roll capture is unchanged. Drum tracks are unaffected (ARP IN is melodic-only).

## [0.3.7] — 2026-05-14
### Fixes
- **Chord-press monitoring now plays every note.** Tapping multiple pads simultaneously (drum or melodic, ROUTE_MOVE or ROUTE_SCHWUNG) used to silently drop notes — same-buffer set_param calls coalesce to last-write-wins regardless of key, so each chord press was reduced to whichever press the host happened to deliver last. Live note events queue synchronously into `pendingLiveNotes` and drain once per tick from `globalThis.tick()` into a single batched `tN_live_notes` payload (`on p1 v1 on p2 v2 ...`). One set_param call per audio buffer = no coalescing = full chord survival. Costs up to ~10 ms of monitor latency on the worst-case-aligned tap, but eliminates the unpredictable dropouts that microtask-per-onMidiMessage scheduling produced when multiple pad CCs landed in one audio buffer.
- **Drum chord recording lands in one DSP buffer.** Queued `_drumRecNoteOns` / `_drumRecNoteOffs` now batch into a single `tN_drum_record_note_on` / `_off` payload per tick (mirroring melodic record batching), instead of trickling one entry per tick via `.shift()`. A 4-pad chord previously took 4 ticks (~42 ms) to fully ingest into DSP; press-time correctness was preserved via `drum_current_step[lane]`, but DSP-side state lagged. DSP `tN_drum_record_note_on` / `_off` parsers extended to loop over `"p1 v1 p2 v2 ..."` and `"p1 p2 ..."` payloads respectively.
- **Drum recording inline-monitors via DSP, matching melodic.** The drum press handler used to fire an extra `queueLiveNoteOn` alongside the record push, producing two same-buffer set_params (`tN_live_notes` + `tN_drum_record_note_on`) that coalesced and dropped notes from armed-track chord recordings. DSP `drum_record_note_on` now calls `live_note_on` per parsed pitch when the track is `ROUTE_MOVE` (mirroring the melodic `record_note_on` pattern at `seq8_set_param.c:3310`); JS no longer emits the redundant monitor set_param. One set_param per chord on the armed path = recording and monitoring ride in the same payload, no collision.

Single tap (no second press before release) keeps its existing semantics: length = `(N+1)*16`, loop_start = 0. Aborted hold (Loop released first) falls back to single-tap semantics so no gesture state leaks. **Non-destructive**: notes outside the window are preserved in storage and replay when you expand the window.
